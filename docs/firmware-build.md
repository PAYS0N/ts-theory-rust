# Firmware build: generated header → `.uf2`

How `out/steno_generated_dictionary_data.h` (built by `build-javelin`, see
[pipeline.md](pipeline.md#the-programmatic-path--expandinfinite)) ends up
inside a flashable `javelin-steno-pico` firmware image. This crosses out of
the Rust workspace entirely — CMake, an ARM cross toolchain, and two vendored
C++ trees — so none of it is exercised by `ctx-verify` or `scripts/ci.sh`.

## Trees involved

| Tree | Role | Tracked here? |
|---|---|---|
| `out/steno_generated_dictionary_data.h`, `out/steno_generated_testdata.h` | Generated by `build-javelin` from `dict.infinite.steno` | No — build product (gitignored) |
| [`javelin-ext/`](../javelin-ext/) | `StenoGeneratedDictionary`, the C++ port of the reference walker (`walk.rs`), consumes the header above | Yes — tracked sibling directory |
| `javelin-steno` | Upstream firmware library ([`jthlim/javelin-steno`](https://github.com/jthlim/javelin-steno)) | No — vendored, gitignored, re-synced |
| `javelin-steno-pico` | Upstream Pico bindings ([`jthlim/javelin-steno-pico`](https://github.com/jthlim/javelin-steno-pico)); `javelin-steno-pico/javelin` is a symlink into `javelin-steno` | No — vendored, gitignored, re-synced |
| `pico-sdk`, `picotool-<version>` | Raspberry Pi's SDK and the `picotool` image-postprocessing tool it depends on | No — vendored, gitignored |

`javelin-ext/` reimplements the walker's lookup semantics independently (see
`README.md`'s Architecture section); it is not itself part of `javelin-steno`
or `javelin-steno-pico`, which is what makes wiring it in a two-step,
cross-repo affair rather than a single edit.

## Why a patch, not an in-place edit

`javelin-steno-pico` isn't part of this repo — it's a vendored checkout,
gitignored and periodically re-synced from upstream. Getting the generated
dictionary into its firmware build needs exactly two edits inside that
checkout:

1. `CMakeLists.txt` — compile `javelin-ext/steno_generated_dictionary.cc` and
   give it the right include paths.
2. `pico_bindings.cc` — register a `StenoGeneratedDictionary` instance into
   the real runtime dictionary list built in `InitJavelinMaster()`.

Editing those files directly would be silently lost the next time
`javelin-steno-pico` is re-synced. So the edits live as a tracked patch
(`scripts/javelin_ext_pico.patch`) reapplied by a script
(`scripts/apply_javelin_ext_patch.sh`), rather than as permanent changes to
a tree this repo doesn't own.

### `scripts/javelin_ext_pico.patch`

A `git diff`-format patch, applied with `git apply` against the
`javelin-steno-pico` checkout. Two hunks:

- `CMakeLists.txt`: adds
  `${CMAKE_CURRENT_SOURCE_DIR}/../javelin-ext/steno_generated_dictionary.cc`
  to `target_sources`, and extends `target_include_directories` with
  `${CMAKE_CURRENT_SOURCE_DIR}/javelin` and `${CMAKE_CURRENT_SOURCE_DIR}/../out`
  — `steno_generated_dictionary.cc` includes `"str.h"` and
  `"dictionary/dictionary.h"` unqualified (matching `cpp_check.sh`'s own `-I`
  flags for the host-side differential test), and needs `out/` for the
  generated data header. `javelin-ext/` itself needs no include entry: its
  own header sits next to the `.cc` and resolves via ordinary
  quoted-include-relative-to-file lookup.
- `pico_bindings.cc`: adds `#include "../javelin-ext/steno_generated_dictionary.h"`,
  and in `InitJavelinMaster()`, registers the dictionary right after the
  user-dictionary registration and before the flash-resident map dictionary
  is added:

  ```cpp
  static StenoGeneratedDictionary generatedDictionary;
  dictionaries.Add(StenoDictionaryListEntry(&generatedDictionary, true));
  ```

  Ordering matters: `StenoDictionaryList::Lookup()`
  (`javelin-steno/dictionary/dictionary_list.cc`) is first-match-by-list-order,
  so a live-edited user dictionary still wins over the generated one, which in
  turn is tried before the bulk uploaded map dictionary. A function-local
  `static` (not `JavelinStaticAllocate`) is correct here because
  `StenoGeneratedDictionary`'s constructor is trivial — it only sets
  `maximumOutlineLength` from the `MAX_OUTLINE_LENGTH` constant baked into the
  generated header — so, unlike the other containers, it has no init-order
  dependency on a flash address read at startup.

  `javelin-ext/test_main.cc` (the host-only differential-test harness,
  `JAVELIN_EXT_RUN_TESTS`-gated) is deliberately **not** added to the firmware
  target.

### `scripts/apply_javelin_ext_patch.sh`

Applies the patch, following `cpp_check.sh`'s SKIP/FAIL conventions:

- SKIP if `javelin-steno-pico/` isn't present (vendored tree not cloned).
- SKIP "already applied" if the marker (`steno_generated_dictionary`) is
  already present in `CMakeLists.txt` — safe to re-run after a fresh clone,
  a partial apply, or as an unconditional step in a build script.
- Otherwise applies the patch (checked first with `git apply --check`, so a
  failure never leaves a half-applied tree); on failure, FAILs with guidance
  toward regenerating the patch, since upstream drift in the surrounding
  lines is the expected failure mode after a `javelin-steno-pico` re-sync:

  ```sh
  # after re-applying the two edits by hand against the freshly synced tree
  git -C javelin-steno-pico diff CMakeLists.txt pico_bindings.cc \
      > scripts/javelin_ext_pico.patch
  ```

This is not wired into `ci.sh`/`ctx-verify`: that gate is scoped to the Rust
workspace, and firmware assembly needs an external ARM toolchain plus a
`-D JAVELIN_BOARD=...` choice, which isn't a pass/fail workspace check.

## Build steps

1. **Generate the header.**

   ```sh
   cargo run --bin build-javelin
   ```

   Writes `out/steno_generated_dictionary_data.h` and
   `out/steno_generated_testdata.h` from `dict.infinite.steno`.

2. **Wire the dictionary into `javelin-steno-pico`.**

   ```sh
   bash scripts/apply_javelin_ext_patch.sh
   ```

3. **Prerequisites**, once per machine (see `javelin-steno-pico/README.md`
   for the canonical version):

   - Arm GNU Toolchain ≥ 12 (this repo has validated against 14.2.1).
   - CMake ≥ 4.
   - `pico-sdk` cloned, with `PICO_SDK_PATH` pointing at it.
   - `javelin-steno` cloned, symlinked as `javelin-steno-pico/javelin`.
   - `picotool`, matching the version `pico-sdk/tools/CMakeLists.txt` pins
     (`picotool_VERSION_REQUIRED`). By default CMake fetches it from
     `https://github.com/raspberrypi/picotool.git` at configure time; on a
     machine without that network access, point CMake at a local clone
     instead:

     ```sh
     cmake .. -D JAVELIN_BOARD=polyglot \
         -D PICOTOOL_GIT_REPOSITORY_URL=/path/to/local/picotool \
         -D PICOTOOL_GIT_BRANCH=<tag matching picotool_VERSION_REQUIRED>
     ```

     (`PICOTOOL_GIT_REPOSITORY_URL` is a CMake variable, not read from the
     environment — unlike `PICOTOOL_FETCH_FROM_GIT_PATH`, which is — so it
     must be passed with `-D`.)

4. **Configure and build**, from within `javelin-steno-pico/`:

   ```sh
   mkdir build && cd build
   cmake .. -D JAVELIN_BOARD=polyglot
   make
   ```

   `polyglot` is an rp2040 board, so no `PICO_PLATFORM` override is needed;
   for rp2350 boards, add `-D PICO_PLATFORM=rp2350-arm-s`. `pico_add_extra_outputs`
   (the last line of `CMakeLists.txt`) produces `javelin-steno-pico.uf2` (plus
   `.elf`, `.bin`, `.hex`) in the build directory — the `.uf2` is what gets
   drag-and-dropped onto the device in BOOTSEL mode.

5. **Re-run after any `javelin-steno-pico` re-sync**: repeat step 2 first —
   if the surrounding lines moved upstream, `apply_javelin_ext_patch.sh`
   FAILs with instructions for regenerating the patch (see above) before the
   configure/build steps can succeed again.

## Known blocker (unrelated to the dictionary wiring)

As of this writing, a full `make` fails to compile for reasons unconnected to
`javelin-ext/`: `javelin/split/split.h` declares an unscoped
`enum SplitMetricId { RESET_COUNT, ... }` at global scope, which collides
with `pico-sdk`'s `hardware/structs/resets.h` (its own `RESET_COUNT` from
`NUM_RESETS`) in every translation unit that transitively includes both —
`pico_bindings.cc`, `ssd1306.cc`, and `st7789.cc` among them, none of which
touch `javelin-ext/`. This looks like a version-skew issue between the
vendored `javelin-steno` and `pico-sdk` checkouts, predating and independent
of this integration.

What *is* validated: `javelin-ext/steno_generated_dictionary.cc` compiles
cleanly to a valid ARM object under the real toolchain, against the real
vendored `javelin-steno` headers and the real generated data header — i.e.
the dictionary-wiring half of this pipeline is confirmed correct. The
`RESET_COUNT` collision fails the *compile* step of those unrelated firmware
translation units (`ssd1306.cc` and `st7789.cc` pull in `split.h` via their
headers while a `hardware/*.h` include drags in `reset_num_t`), never the
generated dictionary's own compilation.
